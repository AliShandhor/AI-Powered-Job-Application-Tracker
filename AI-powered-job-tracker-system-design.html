Section 01

System Architecture

The system follows a clean, layered architecture with strict separation of concerns. The frontend is a statically optimized Next.js app with server components. The backend is a FastAPI service exposing a versioned REST API. An AI worker process handles async intelligence tasks off the critical path.

Next.js Frontend
React Server Components for initial page load, client components for interactivity. Deployed to Vercel edge network. Communicates only with the API gateway.

Section 02

Tech Stack Justification

Each technology choice is deliberate. Here's the rationale for every major decision and what was considered but rejected.

Frontend Framework
Next.js 14 (App Router)
Server Components reduce JS bundle by ~60%. Built-in image optimization, code splitting, and Vercel deployment make it production-ready out of the box. App Router enables granular caching strategies per route.
Rejected: Remix (smaller ecosystem), Vite SPA (no SSR, worse SEO/performance)
Backend Framework
FastAPI (Python)
Async-native, automatic OpenAPI docs, excellent type safety via Pydantic v2. The Python ecosystem is essential for AI/ML integrations (LangChain, OpenAI SDK, spaCy). 3x faster than Flask/Django for I/O-bound tasks.
Rejected: Django (sync overhead), Express (weaker AI ecosystem), Rails (wrong language for AI)
Primary Database
PostgreSQL 16
Row-Level Security for multi-tenancy at the DB layer — the safest approach for SaaS data isolation. JSONB columns for flexible job posting metadata. Full-text search via pg_trgm. Mature, battle-tested.
Rejected: MySQL (weaker JSON support, no RLS), MongoDB (overkill flexibility, worse joins)
Authentication
Clerk
Complete auth solution with prebuilt UI components, OAuth providers, magic links, MFA, and session management. Eliminates auth surface area from our codebase. JWTs are verifiable without network calls.
Rejected: Auth0 (more expensive), Supabase Auth (vendor lock-in to Supabase stack), custom JWT (security risk)
Task Queue
Celery + Redis
Industry-standard Python async task queue. Redis serves dual duty as the message broker and result backend. Retry logic, task routing, and monitoring (Flower) built in. Scales to multiple workers trivially.
Rejected: RQ (less features), Dramatiq (smaller ecosystem), SQS (adds AWS dependency)
Vector Search
Pinecone
Managed vector database for job-to-resume similarity search. Serverless tier is cost-efficient at low volume. Sub-millisecond ANN queries. Handles embedding storage and retrieval so we don't hack this into PostgreSQL.
Rejected: pgvector (slower at scale), Weaviate (self-hosted complexity), Qdrant (less managed tooling)

Row-Level Security Policies

migrations/002_rls_policies.sql SQL 
-- Enable RLS on all user-owned tables
ALTER TABLE applications ENABLE ROW LEVEL SECURITY;
ALTER TABLE events ENABLE ROW LEVEL SECURITY;
ALTER TABLE resumes ENABLE ROW LEVEL SECURITY;

-- Policy: users only see their own applications
CREATE POLICY applications_isolation ON applications
  USING (user_id = current_setting('app.current_user_id')::uuid);

-- Policy: events isolated via application ownership
CREATE POLICY events_isolation ON events
  USING (application_id IN (
    SELECT id FROM applications
    WHERE user_id = current_setting('app.current_user_id')::uuid
  ));

-- Set user context on connection (called in middleware)
SELECT set_config('app.current_user_id', :user_id, true);

Section 04

API Design

RESTful API versioned under /api/v1. All endpoints require a valid Clerk JWT in the Authorization header. Responses follow a consistent envelope schema. Errors return RFC 7807 Problem Details.

Applications

GET /applications List all applications for authenticated user. Supports filtering by status, date range, and full-text search. Returns paginated results with cursor-based pagination.
POST /applications Create a new application. Triggers async AI job: extract key requirements from job description and compute initial match score against user's primary resume.
GET /applications/{id} Fetch single application with full timeline, contacts, and AI insights. Includes cached AI analysis results if available.
PATCH /applications/{id} Partial update. Status transitions are validated against a state machine (applied → screening → interview → offer → accepted/rejected). Invalid transitions return 422.
DELETE /applications/{id} Soft-delete (sets deleted_at). Hard delete via admin endpoint only. Cascades to events and contacts.
Events & Contacts
POST /applications/{id}/events Add a timeline event (interview scheduled, offer received, follow-up sent). Triggers reminder creation if event_date is in the future.
GET /applications/{id}/contacts List contacts associated with this application (recruiters, hiring managers, interviewers).
POST /applications/{id}/contacts Add a contact. Optionally triggers AI enrichment to find their LinkedIn or email from public sources.
AI Endpoints
POST /ai/analyze-job Submit a job description for AI analysis. Returns a task_id. Async: extracts required skills, infers seniority, identifies company culture signals.
POST /ai/match-score Compute semantic similarity between a resume and a job description using embeddings. Returns score 0-100 with detailed breakdown by skill category.
POST /ai/interview-prep Generate tailored interview questions and suggested answers based on the job description and the user's resume. Streamed response via SSE.
POST /ai/cover-letter Generate a tailored cover letter. Streamed response. User provides tone preference and highlights to emphasize.
GET /ai/tasks/{task_id} Poll async AI task status. Returns pending | processing | complete | failed with partial or full result when available.
Resumes
POST /resumes/upload Upload a resume PDF. Extracts text via pdfplumber, runs AI parsing to structure skills/experience, generates and stores embedding in Pinecone.
GET /resumes List all resume versions for the user. Users can store multiple tailored resumes.

FastAPI Application Factory
The app factory pattern keeps the application instance clean and testable. Middleware is registered here in the correct order (CORS before auth before rate limiting).
app/main.py Python 
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from contextlib import asynccontextmanager
from app.core.config import settings
from app.core.database import engine
from app.domains.applications.router import router as app_router
from app.domains.ai.router import router as ai_router

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup: validate DB connection, warm caches
    async with engine.begin() as conn:
        await conn.execute(text("SELECT 1"))
    yield
    # Shutdown: close pool gracefully
    await engine.dispose()

def create_app() -> FastAPI:
    app = FastAPI(
        title="JobTracker AI",
        version="1.0.0",
        lifespan=lifespan,
        docs_url="/api/docs" if settings.DEBUG else None,
    )
    app.add_middleware(
        CORSMiddleware,
        allow_origins=settings.CORS_ORIGINS,
        allow_credentials=True,
        allow_methods=["*"],
        allow_headers=["*"],
    )
    app.include_router(app_router, prefix="/api/v1")
    app.include_router(ai_router, prefix="/api/v1")
    return app

app = create_app()
03
Authentication Middleware
Validate Clerk JWTs on every request. Set the Postgres RLS context so the database enforces data isolation automatically — even if application code has bugs.
app/core/deps.py Python 
from fastapi import Depends, HTTPException, status
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
from jose import jwt, JWTError
from sqlalchemy.ext.asyncio import AsyncSession
from app.core.database import get_db_session
from app.domains.users.repo import UserRepo

security = HTTPBearer()

async def get_current_user(
    credentials: HTTPAuthorizationCredentials = Depends(security),
    db: AsyncSession = Depends(get_db_session),
):
    try:
        payload = jwt.decode(
            credentials.credentials,
            settings.CLERK_JWT_PUBLIC_KEY,
            algorithms=["RS256"],
        )
        clerk_id: str = payload.get("sub")
        if not clerk_id:
            raise HTTPException(status.HTTP_401_UNAUTHORIZED)
    except JWTError:
        raise HTTPException(status.HTTP_401_UNAUTHORIZED)

    user = await UserRepo(db).get_by_clerk_id(clerk_id)
    if not user:
        raise HTTPException(status.HTTP_404_NOT_FOUND, "User not found")

    # Set RLS context — Postgres will enforce isolation
    await db.execute(
        text("SELECT set_config('app.current_user_id', :uid, true)"),
        {"uid": str(user.id)},
    )
    return user

Domain Service Layer
The service layer owns business logic. It's the only layer that knows about both HTTP schemas and database models. Keep routers thin — they translate HTTP concerns only.
app/domains/applications/service.py Python 
from app.domains.applications.repo import ApplicationRepo
from app.domains.applications.schemas import ApplicationCreate, ApplicationOut
from app.workers.tasks.ai_tasks import analyze_job_async
from app.core.exceptions import NotFoundError, ForbiddenError
import uuid

class ApplicationService:
    STATUS_MACHINE = {
        "applied":    {"screening", "rejected", "withdrawn"},
        "screening":  {"interview", "rejected"},
        "interview":  {"offer", "rejected"},
        "offer":      {"accepted", "rejected", "negotiating"},
        "negotiating":{"accepted", "rejected"},
    }

    def __init__(self, repo: ApplicationRepo):
        self.repo = repo

    async def create(self, user_id: uuid.UUID, data: ApplicationCreate) -> ApplicationOut:
        app = await self.repo.create(user_id, data)
        # Enqueue AI analysis — non-blocking
        if data.job_description:
            analyze_job_async.delay(str(app.id), data.job_description)
        return ApplicationOut.model_validate(app)

    async def transition_status(self, app_id, current_user_id, new_status):
        app = await self.repo.get_or_404(app_id)
        allowed = self.STATUS_MACHINE.get(app.status, set())
        if new_status not in allowed:
            raise ValueError(
                f"Cannot transition from {app.status} to {new_status}"
            )
        return await self.repo.update_status(app_id, new_status)

Type-Safe API Client
A thin typed wrapper around fetch — no Axios, no code generation overhead for this scale. Every response is typed and errors are handled uniformly.
frontend/lib/api.ts TypeScript 
import { auth } from "@clerk/nextjs/server";

const BASE = process.env.NEXT_PUBLIC_API_URL;

async function apiFetch<T>(
  path: string,
  init?: RequestInit
): Promise<T> {
  const { getToken } = auth();
  const token = await getToken();

  const res = await fetch(`${BASE}${path}`, {
    ...init,
    headers: {
      "Content-Type": "application/json",
      Authorization: `Bearer ${token}`,
      ...init?.headers,
    },
    next: { revalidate: 0 },
  });

  if (!res.ok) {
    const err = await res.json().catch(() => ({}));
    throw new ApiError(res.status, err.detail ?? "Unknown error");
  }

  const body = await res.json();
  return body.data as T;
}

export const api = {
  applications: {
    list: (params?) => apiFetch(`/api/v1/applications?${new URLSearchParams(params)}`),
    get: (id: string) => apiFetch(`/api/v1/applications/${id}`),
    create: (body) => apiFetch(`/api/v1/applications`, { method: "POST", body: JSON.stringify(body) }),
    patch: (id, body) => apiFetch(`/api/v1/applications/${id}`, { method: "PATCH", body: JSON.stringify(body) }),
  },
};

Section 06

AI Feature Implementation

Four production AI features, each designed to deliver measurable value. All are async and gracefully degrade — if the AI service is down, the core app still functions.

🎯
Job–Resume Match Scoring
Generates text-embedding-3-large embeddings for both the job description and the user's resume. Computes cosine similarity in Pinecone. Returns a 0–100 score with per-skill-category breakdown (technical skills, soft skills, experience level, domain fit). Scores update automatically when the user uploads a new resume. Average latency: 800ms on the worker; zero latency on the API (async).
📋
Resume & JD Parsing
Uses GPT-4o with structured output (JSON mode) to extract skills, years of experience per domain, education, company names, and employment gaps from raw resume text. For job descriptions, extracts required vs. preferred qualifications, seniority signals, compensation signals, and company culture markers. Results stored in JSONB for fast re-querying without re-inference.
💬
Interview Preparation
Generates tailored interview question sets streamed via Server-Sent Events. System prompt is constructed from the parsed job requirements + user's resume + company research context. Output format: 10 behavioral questions (STAR-formatted suggested answers), 5 technical questions with hints, and 3 questions for the candidate to ask. Streamed to avoid timeout on long generation.
✍️
Cover Letter Generation
Streamed GPT-4o generation with a structured system prompt that enforces: no clichés ("I am excited to apply..."), specificity requirement (must reference actual job requirements), and a chosen tone (formal, conversational, or startup-casual). The prompt includes the full job description, user's resume, and 2-3 user-provided highlights to emphasize. Users can regenerate with modified parameters without another API call — uses client-side prompt rebuilding.

from celery import shared_task
from openai import AsyncOpenAI
from app.workers.celery_app import celery
from app.core.database import get_sync_session
from app.domains.ai.service import AITaskService

oai = AsyncOpenAI()

@celery.task(
    bind=True,
    queue="ai",
    max_retries=3,
    autoretry_for=(Exception,),
    retry_backoff=True,
    retry_backoff_max=120,
)
def analyze_job_async(self, application_id: str, job_description: str):
    with get_sync_session() as db:
        svc = AITaskService(db)
        svc.set_task_processing(application_id)

        response = oai.chat.completions.create(
            model="gpt-4o",
            response_format={"type": "json_object"},
            messages=[
                {"role": "system", "content": JOB_ANALYSIS_PROMPT},
                {"role": "user", "content": job_description},
            ],
            temperature=0.1,  # Low temp for structured extraction
        )

        result = json.loads(response.choices[0].message.content)
        svc.store_job_analysis(application_id, result)
        svc.set_task_complete(application_id, result)

JOB_ANALYSIS_PROMPT = """
Extract structured data from this job description. Return JSON with keys:
- required_skills: list of strings (technical skills explicitly required)
- preferred_skills: list of strings (nice-to-have)
- seniority_level: "junior"|"mid"|"senior"|"staff"|"principal"
- estimated_yoe: integer (years of experience implied)
- domain: primary industry/domain string
- compensation_signals: object with min/max/currency if present
- culture_signals: list of strings (remote/hybrid/startup/etc)
- key_requirements: top 5 most important requirements as bullet strings
"""

SSE Streaming Endpoint

app/domains/ai/router.py Python 
from fastapi import APIRouter
from fastapi.responses import StreamingResponse

@router.post("/ai/interview-prep")
async def interview_prep(
    req: InterviewPrepRequest,
    user = Depends(get_current_user),
):
    async def stream_response():
        async with oai.chat.completions.stream(
            model="gpt-4o",
            messages=build_interview_messages(req, user),
        ) as stream:
            async for chunk in stream:
                delta = chunk.choices[0].delta.content or ""
                yield f"data: {json.dumps({'delta': delta})}\n\n"
        yield "data: [DONE]\n\n"

    return StreamingResponse(
        stream_response(),
        media_type="text/event-stream",
        headers={"X-Accel-Buffering": "no"},
    )

Infrastructure

Frontend — Vercel

Connect GitHub repo. Set environment variables. Automatic preview deployments on every PR. Production deploys from main branch only. Enable Vercel Analytics and Speed Insights. Custom domain via Cloudflare CNAME.

Backend API — Railway

Deploy from Dockerfile. Railway provisions Postgres and Redis in the same private network — zero-latency internal connections. Scale API service to 2 replicas minimum. Set health check to /health.

Celery Workers — Railway

Separate Railway service running celery worker -Q ai,default. Scale AI worker to 1 replica (GPT-4o rate limits make more workers wasteful). Default queue can scale to 3. Monitor via Flower dashboard.

CDN / Security — Cloudflare

Proxy all traffic. WAF rules: block common attack patterns, rate-limit auth endpoints to 10 req/min/IP. Cache static assets at edge. Add security headers (HSTS, CSP, X-Frame-Options) via Transform Rules.

Scaling Strategy

0–1K users (MVP): Single API replica, 1 Celery worker per queue, Railway Starter plan, free Pinecone tier. Cost: ~$25/month.

1K–10K users: 2 API replicas, auto-scale Celery AI worker to 2, add PgBouncer, Redis cluster. Add Cloudflare Workers for edge caching of AI results. Cost: ~$200/month.

10K+ users: Migrate API to GCP Cloud Run (per-request billing), Celery on GKE with KEDA autoscaling, Cloud SQL with read replicas, regional Cloudflare deployment. Instrument everything with OpenTelemetry → Grafana.
